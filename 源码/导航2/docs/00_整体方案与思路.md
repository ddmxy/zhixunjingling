# RDK_Navigation 整体方案与思路

> 目标：基于 **RDK X5** 开发板做一台 2D 室内移动机器人，实现 **建图 → 保存地图 → RViz 中点选目标 → 自动导航避障** 的完整闭环。
>
> 下位机（STM32F103）已经写好运动控制和数据上报；本项目要做的是 **上位机（RDK X5 上的 ROS 2 工作空间）**：把下位机当成一个"传感器+执行器"，把激光雷达当成"另一个传感器"，把它们都接进 ROS 2 → SLAM → Nav2 这条标准管线。

### 1.1 下位机给上位机发什么、上位机给下位机发什么

详细字节级协议见 `01_下位机串口协议.md`，这里先给一个高层概览：

| 方向 | 内容 | 频率 | 备注 |
|---|---|---|---|
| ↑上行 | 车体线速度 `vx`、`vy` (m/s) | 10 Hz | TIM5 50 ms × 交替，差速车 `vy=0` |
| ↑上行 | 偏航角 `yaw` (rad)、角速度 `w` (rad/s) | 10 Hz | yaw ∈ [-π, π]，IMU 已减零偏 |
| ↓下行 | 目标线速度 `target_v` (m/s)、目标角速度 `target_w` (rad/s) | 跟 `/cmd_vel` 同步 | 一收到立即下发 |

**关键点**：
- 下位机自己**不带时间戳**。所有时间戳都由上位机在收到字节流的瞬间用 `rclcpp::Clock::now()` 打上去。
- 下位机的位姿（x, y, θ）**不在协议里**，需要上位机自己对 `vx + w`（或 `vx + yaw`）做积分。

### 1.2 雷达侧

- 型号：**LSLIDAR N10**（资料：`D:\Ksoftware\32\N10系列激光雷达附送资料\N10系列雷达客户资料V5.0_20240822\2.ROS2_SDK\LSLIDAR_X_ROS2-20240228`）
- 接口：USB → TTL 串口（厂商带的转换板），Linux 下默认设备节点 `/dev/wheeltec_laser`（依赖 udev 规则）
- ROS 2 节点：`lslidar_driver_node`，发布 `sensor_msgs/LaserScan` → `/scan`，`frame_id = laser`
- 测距范围：0.2 ~ 25 m（N10 实际有效约 12 m，配置文件里写到 200 m 是上限保护）
- 频率：6~12 Hz（看雷达型号和波特率）

雷达 SDK 直接整体拷贝到 `ros2_ws/src/`，**不需要改源码**，只改 launch / yaml。

### 2.2 关键 TF 树

```
map ──► odom ──► base_footprint ──► base_link ──┬─► laser
                                                 ├─► imu_link
                                                 └─► (4 个轮子，可选)
   ▲          ▲                ▲                 ▲
   │          │                │                 │
slam_toolbox  base_driver  base_driver       URDF static
```

- `map → odom`：由 `slam_toolbox`（建图时）或 `amcl`/slam_toolbox localization（导航时）发布
- `odom → base_footprint`：由 **`base_driver` 自己** 发布（积分 vx, w 而来）
- `base_footprint → base_link → laser/imu_link`：URDF 描述的固定关系，由 `robot_state_publisher` 自动发

## 3. base_driver 节点的内部逻辑（重点）

这是上位机最核心、也是必须自己写的部分。把它的职责写清楚：

### 3.1 启动时

1. 打开串口（参数化：`/dev/ttyUSB0`, `115200`, `8N1`）
2. **上行回传依赖下行握手**：下位机收到**第一帧合法** `0xFF 0x02 [v][w] 0xFE` 后才会置 `start_Flag` 并开始周期发 `0x03`/`0x04`。因此 `base_driver` 在打开串口后应**立即发送一帧** `v=0, w=0`（与正常控制帧格式完全相同）；若长时间无 `/cmd_vel`，可低速周期发零速帧以维持上报（可选，见协议文档参数）。
3. 启动一个**接收线程**：阻塞读字节、按状态机解析下位机的 `0xFF 0x03/0x04 ... 0xFE` 帧
4. 注册 `/cmd_vel` 订阅、`/odom` `/imu/data` 发布、TF 广播

### 3.2 接收解析（状态机：解析上行）

上位机这边的状态机只对 **MCU → PC** 的 `0x03` / `0x04` 帧有效。

> 方向不要混：**PC → MCU** 的帧在 MCU 上用 `USART1_IRQHandler` 解析，`0xFF` 后第二字节**只能是 `0x02`**，再接 8 字节与尾 `0xFE`，与上位机解析上行的 `0x03` / `0x04` **不是同一套**。见 `docs/01_下位机串口协议.md` §4.0。

```
state 0 : 等 0xFF
state 1 : 看 cmd 字节 → 0x03=速度包 / 0x04=偏航包
state 2 : 收 8 字节 payload
state 3 : 校验尾字节 0xFE → 触发回调
```

收到完整帧后：

- **0x03 速度包**：`memcpy` 出两个 little-endian float → `vx`, `vy`（m/s）
- **0x04 偏航包**：`memcpy` 出 → `yaw_rad`, `w_rad/s`

每收到一帧，立刻 `t_now = node->now()` 打时间戳。

### 3.3 状态融合 / 位姿积分

下位机各 10 Hz 交替发两种包，所以最新的 `vx` 和最新的 `yaw/w` 时间戳差最多 50 ms。处理方式：

- **维护一个最新状态**：`(vx, vy, yaw, w, t_last)`，每收到任一帧就更新对应字段。
- **在固定速率定时器（建议 50 Hz）里发布 `/odom`**：
  - `theta = yaw`（**直接用 IMU 的 yaw**，不要轮速积分 θ — 因为打滑/编码器误差累积很快，IMU yaw 短时间内更准）
  - 用 `dt = t_now - t_last_publish` 积分位置：
    ```
    x += vx * cos(theta) * dt - vy * sin(theta) * dt
    y += vx * sin(theta) * dt + vy * cos(theta) * dt
    ```
  - twist：`linear.x = vx`, `linear.y = vy`, `angular.z = w`
- **同时发 TF**：`odom → base_footprint`，平移 `(x, y, 0)`，旋转 `(0,0, sin(θ/2), cos(θ/2))`
- 协方差：`pose.covariance` 和 `twist.covariance` 给一个经验对角阵（可参数化）

### 3.4 IMU 发布（可选但建议）

把 0x04 包里的 yaw 同步发一份 `sensor_msgs/Imu` 到 `/imu/data`：
- `orientation` ← `yaw → quaternion`
- `angular_velocity.z` ← `w`
- `linear_acceleration` 留空（下位机当前协议没传，下一版可加）

未来要做 robot_localization (EKF) 时会用到。

### 3.5 cmd_vel 下发

订阅 `geometry_msgs/Twist`：
- `target_v = msg.linear.x`
- `target_w = msg.angular.z`
- 打包 `0xFF 0x02 [v float LE] [w float LE] 0xFE`（**共 11 字节**：`SOF + CMD + 8 B payload + EOF`），写串口
- 加一个**看门狗**：超过 0.5 s 没收到 cmd_vel 就主动发 (0,0)，防止 Nav2 崩了车还在跑

## 4. 工作流（用户的实际操作流程）

把整个项目拆成 3 个独立可运行的"模式"，每个模式都对应一个 launch 文件。

### 4.1 模式 A：建图（`car_bringup mapping.launch.py`）

```
[雷达驱动] + [base_driver] + [robot_state_publisher] + [slam_toolbox: online_async] + [rviz2]
```
操作：
1. 在 RViz 里看着 `/scan` 和 `/map`，**手柄/键盘遥控小车**走一圈整个场地
2. `ros2 run nav2_map_server map_saver_cli -f ~/maps/my_room` 保存地图

> 遥控用 `teleop_twist_keyboard` 或 `joy + teleop_twist_joy`，发布 `/cmd_vel`。

### 4.2 模式 B：导航（`car_bringup navigation.launch.py map:=...`）

```
[雷达驱动] + [base_driver] + [robot_state_publisher] + [Nav2] + [AMCL or slam_toolbox: localization] + [rviz2]
```
操作：
1. 在 RViz 里用 **2D Pose Estimate** 给一个初始位姿
2. 用 **2D Nav Goal** 点击地图任意位置 → Nav2 自动规划路径 → 输出 `/cmd_vel` → 小车走过去
3. 中途有动态障碍物时，局部 costmap 会避障

### 4.3 模式 C：调试 / 单独跑（`car_bringup driver_only.launch.py`）

只起雷达 + base_driver + RSP + RViz，不跑 SLAM/Nav，用来：
- 看 `/scan` 是否正常
- 看 `/odom` 在原地推车时是否合理（推 1 m 看 x 是否大约增 1 m）
- 看 `/tf` 树是否完整无断点

## 6. 已确认参数基线（2026-05-08）

以下配置已由你确认，后续代码和参数将按这份基线实现：

1. **STM32 ↔ RDK X5 接法**
   - 使用 **USB-TTL 转换线**，上位机串口按 `/dev/ttyUSB*` 处理（默认 `/dev/ttyUSB0`，可参数改）。

2. **N10 雷达接法**
   - 使用雷达自带 USB 转串口方案，按 `/dev/wheeltec_laser` 接入（基于厂商 udev 规则）。

3. **运动学类型**
   - 车辆按**差速车**处理（不能侧向平移，像汽车一样），`vy=0`。
   - 导航局部控制器选 **RPP (Regulated Pure Pursuit)**。

4. **雷达安装位姿（当前阶段）**
   - 先按**车体中心安装**处理，角度先留参数接口，后续你实测后改。
   - 高度按 `z = 0.120 ~ 0.130 m` 先占位，URDF 默认先写 `z=0.125 m`。

5. **车体参数（你给定的机械参数）**
   - 车体尺寸：`L=244.6 mm, W=184.7 mm, H=120 mm`
   - 轮子直径：`65 mm`
   - 减速比：`29:1`
   - 编码器：`13 PPR`
   - 说明：后续 `base_driver` 中会把这些都做成 ROS 参数，便于在线微调。

6. **建图方案**
   - 使用 `slam_toolbox`，后续按场景做参数适配。

7. **建图遥控方式**
   - 使用键盘 `teleop_twist_keyboard`。
   - 你习惯的按键为 `i / m / < / >`（前进、后退、原地转向），launch 中会保留默认键位提示。

## 7. 风险与已知坑

| 风险 | 表现 | 缓解措施 |
|---|---|---|
| 串口字节序 | 上位机解出来的 float 是 NaN/极大值 | float 是 little-endian，跟 STM32 默认一致。先打印 hex 字节再 cast 验证 |
| 未下行握手 | 始终收不到 `0x03`/`0x04` | 见 §3.1：打开串口后先发一帧合法 `0xFF 0x02` + 两 float + `0xFE`（可先 `v=w=0`）置位 `start_Flag` |
| 下行 CMD 写错 | 下位机不更新目标、不报里程 | MCU 只吃第二字节 **`0x02`**；误发 `0x03`/`0x04` 会在 State1 被拒，整帧丢掉 |
| 时间戳错位 | 0x03 和 0x04 差 50 ms，odom 抖 | base_driver 用最新值，固定 50 Hz 输出 odom 即可，slam_toolbox 自带 ICP 容忍 |
| IMU yaw 零漂 | 小车不动 yaw 也在变 | 启动前静止 3s，主程序里给 IMU 发了 `0xFF 0xAA 0x69 ... + 0xFF 0xAA 0x01 0x08 0x00` 命令做加计校准/零偏 |
| TF 时间不同步 | RViz 报 "TF_OLD_DATA" | 全部使用 `node->now()`；必要时设 `use_sim_time:=false` |
| Nav2 cmd_vel 节流 | 控制器频率太高串口塞车 | base_driver 内部限频到 20 Hz 下发，cmd_vel 突变时合并最新值 |
| 4 轮编码器符号 | 推车 odom 反向 | `Car.h` 中 `CAR_ENCODER_SIGN_W*` 已配，但若联调反向需在 base_driver 端加 `vx_sign` 参数兜底 |
